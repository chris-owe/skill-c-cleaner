## ❌ 反面案例（Anti-Patterns）

> ⚠️ **这些是反面教材！请勿模仿！**
> 
> 通过展示**错误用法**及其后果，帮助您避坑。

### 反面案例 1: 🔇 一刀切全量扫描（低效且浪费）
**文件**: [negative/case-bad-01-blind-scan.md](negative/case-bad-01-blind-scan.md)

| 属性 | 详情 |
|------|------|
| **场景** | 新手用户不分情况就要求"全面扫描" |
| **问题行为** | 直接运行12类别全扫，不经过预检和策略选择 |
| **后果** | 浪费10分钟+10K tokens，信息过载，不知道重点 |
| **正确做法** | 应该先用快速预检(15秒)，再决定扫描范围 |
| **浪费资源** | 时间：多花8分钟 / Token：多花7K (¥0.05) |

**❌ 错误示范**:
```
用户: "帮我全面扫描一下C盘，越详细越好"
AI: [沉默5分钟...开始12类别全量扫描...又5分钟]
    "报告生成完毕！（200页）"
    
结果:
✗ 用户等了10分钟（其实只需要2分钟）
✗ Token消耗10K（其实4.5K就够了）
✗ 报告太长找不到重点（80%的信息与用户无关）
✗ 用户感到困惑："这么多信息，我先看哪个？"
```

**✅ 正确做法**:
```
用户: "帮我看一下C盘"
AI: [15秒快速预检] "检测到你的情况...
    推荐 ⚡ 快速模式（2分钟），因为：
    • 你的C盘还有45%空间，不需要紧急处理
    • 主要占用是开发缓存，重点扫C类即可
    
    要继续吗？"  ← 让用户决定
```

**💡 核心教训**:
> **不要假设所有用户都需要'最详细'的分析。**
> **先用15秒预检了解情况，再推荐合适的方案。**

### 反面案例 2: ⚠️ 忽略AI建议盲目操作（危险！）
**文件**: [negative/case-bad-02-ignore-advice.md](negative/case-bad-02-ignore-advice.md)

| 属性 | 详情 |
|------|------|
| **场景** | 用户看到"可删除"标记就直接删，不看风险提示 |
| **问题行为** | 跳过AI的风险评估，不读理由，机械执行 |
| **后果** | 删除重要缓存导致工作流中断，或删除后才发现需要重建 |
| **正确做法** | 理解每条建议的理由，评估对自己的影响 |
| **风险等级** | 🔴 中等（可能导致数据丢失或工作中断）|

**❌ 错误示范**:
```
AI: "检测到 npm 缓存 3.2GB：
    ⚠️ 注意：你上周编译过项目X，如果近期还要开发，
          建议保留7天后再清理。
    
    其他选项：
    - 如果设置 npm cache 到 D 盘，可以一劳永逸"

用户: "不管了，全部删！"  ← ❌ 忽略警告
[执行清理]

3天后:
用户: "糟糕！项目X编译失败了，node_modules依赖找不到了！"
      "重新 npm install 花了20分钟，耽误了发布会..."
```

**✅ 正确做法**:
```
用户: "npm 缓存能删吗？为什么有警告？"
AI: "可以删，但要注意：
    ✅ 安全性：100%可恢复（npm install 会重新下载）
    ⚠️ 便利性：首次 install 会变慢（约10-20分钟）
    
    我的建议：
    • 如果你本周不再做 Node.js 项目 → 可以安全删除
    • 如果你马上要发布 → 建议保留，或者先迁移到 D 盘
    
    你最近会用到 Node.js 吗？"

用户: "下周才开始新项目"
AI: "那可以安全删除！另外建议你设置环境变量：
    npm config set cache 'D:\\DevCaches\\npm'
    这样以后就不会再占C盘了"
[用户理解后执行，并配置了环境变量]
```

**💡 核心教训**:
> **AI 的价值不只是告诉你'怎么做'，更是解释'为什么'。**
> **跳过解释环节 = 放弃 AI 的核心优势 = 回到传统工具的水平。**

## 📊 案例对比矩阵

| 维度 | 正面案例特征 | 反面案例特征 |
|------|------------|-------------|
| **沟通方式** | 双向对话，互相确认 | 单向指令，机械执行 |
| **决策过程** | 先预检→再选择→后执行 | 直接执行，无策略思考 |
| **风险控制** | 读取警告，理解后果 | 忽略警告，追求速度 |
| **效率观念** | 选择最适合的模式（非最全）| 追求"最详细"（过度）|
| **成本意识** | 关注 Token 消耗和时间 | 不关心资源浪费 |
| **学习态度** | 追问"为什么"，理解原理 | 只关心"怎么删" |
| **结果满意度** | 高（精准解决问题）| 低（要么浪费时间，要么出问题）|

## 🎓 从案例中提炼的核心原则

### 原则 1: 🤝 信任但验证

```
✅ 好: "AI 说可以删，但我先问清楚为什么"
❌ 差: "AI 说可以删，那就删吧"
```

### 原则 2: ⏱️ 时间敏感度匹配

```
✅ 好: 紧急状态用极速模式（30秒），月度体检用标准模式（5分钟）
❌ 差: 不管什么情况都用标准模式（浪费时间）
```

### 原则 3: 💰 成本意识

```
✅ 好: "我只想查 Docker，用自定义模式省 75% 成本"
❌ 差: "给我最详细的报告"（可能 90% 信息无用）
```

### 原则 4: 🎯 聚焦核心矛盾

```
✅ 好: "我的问题是 Docker 太大，其他不用管"
❌ 差: "帮我全面分析"（分散注意力）
```

### 原则 5: 📖 理解 > 执行

```
✅ 好: "为什么要迁移而不是删除？"
❌ 差: "直接给我迁移命令"（不知道为什么要做）
```
